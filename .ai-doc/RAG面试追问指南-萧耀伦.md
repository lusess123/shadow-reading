# 🔥 RAG 项目深度面试追问指南（萧耀伦 - 后端开发）

> 针对简历中"面向内部研发的 RAG 智能知识库与问答系统"项目，设计的完整追问链路。
> 每个问题带有**标准答案 + 判断标准 + 追问陷阱**，方便面试官现场使用。

---

## 📖 30 秒搞懂 RAG

> **RAG = Retrieval-Augmented Generation**（检索增强生成）
>
> 简单说就是：用户问问题 → 先从知识库里**搜出相关文档** → 把文档塞给 LLM → LLM **基于这些文档**来回答
>
> 好处：LLM 不再瞎编（幻觉），因为它是"看着资料答题"

### RAG 完整链路

```
原始文档(PDF/Confluence/Git) → 解析 → 结构化文本 → 切片(Chunks)
    → 向量化 → 向量数据库 → 检索召回 → 重排序 → LLM生成 → 质量评估
```

---

## 📊 关键指标速查表

| 指标 | 通俗解释 | 考察对象 |
|------|---------|---------|
| **Context Recall**（上下文召回率） | 该找到的资料，找全了吗？ | 考察**检索**能力 |
| **Context Precision**（上下文精确率） | 找到的资料里，有多少是真正有用的？ | 考察**检索**能力 |
| **Faithfulness**（忠实度） | LLM 的回答有没有瞎编？是不是都基于检索到的文档？ | 考察**生成**能力 |
| **Answer Relevance**（答案相关性） | LLM 的回答切题吗？有没有答非所问？ | 考察**生成**能力 |

> **Context Recall 通俗类比**：老师出题"Python 的 GIL 是什么？"，标准答案有 3 个关键点，学生翻书找到了 2 个 → Recall = 2/3 = 66.7%。就是"该找到的资料找全了没"。

## 🔗 阶段一：数据管道（解析 + 切片）

### Q1：PDF 解析

> **"你简历里提到用 Unstructured 做 PDF 解析，能说说具体遇到了什么问题吗？多栏排版和表格丢失你是怎么处理的？"**

#### ✅ 好的回答应该包含：

- **多栏问题**：PDF 本身不存储"段落"信息，只存字符坐标。多栏 PDF 如果直接提取文本，会把左栏和右栏的文字混在一起（比如左边第一行接右边第一行），语义完全乱了
- **Unstructured 的解法**：使用 `hi_res` 策略（基于布局检测模型，如 detectron2/YOLOX），先识别页面布局（标题、段落、表格区域），再按阅读顺序提取文本
- **表格处理**：Unstructured 会把表格识别出来，可以用 `table_as_html` 参数保留表格结构，或者转成 Markdown 表格格式

#### 🚩 判断标准：

| 回答水平 | 表现 |
|---------|------|
| ✅ 真做过 | 能说出 `hi_res` vs `fast` 策略的区别，能讲出具体踩过的坑（比如某些 PDF 扫描件要先 OCR） |
| ⚠️ 半懂 | 只说"用了 Unstructured 就解决了"，说不清具体怎么配置 |
| ❌ 背的 | 完全答不出细节，或者把 Unstructured 和 LangChain 的 PyPDFLoader 搞混 |

#### 🎯 追问陷阱：

> "如果 PDF 是扫描件（图片型），Unstructured 能处理吗？"

**答案**：可以，需要开 OCR（`strategy="hi_res"` + 底层调 Tesseract 或 PaddleOCR），但精度会下降，特别是中文文档。如果他说"不能"或者没概念，说明没真正处理过多样化文档。

### Q2：切片策略

> **"你说用 Markdown 层级感知切片比传统定长切分好 40%，这个层级感知具体是怎么做的？"**

#### ✅ 好的回答应该包含：

- **传统定长切分（RecursiveCharacterTextSplitter）**：按固定字数（比如 500 字）暴力切，不管语义边界。经常把一个完整段落切断，或者把标题和内容分开
- **层级感知切片的做法**：
  - 先按 Markdown 标题层级（`#`、`##`、`###`）作为主要分割点
  - 每个 chunk 自动**继承父级标题**作为上下文（比如切出一段内容，会自动把 `# 部署指南 > ## Docker 配置` 这样的路径加到 chunk 开头）
  - 这样即使 chunk 被单独检索出来，LLM 也知道这段话属于什么章节
- **为什么好 40%**：因为搜索的时候，带了标题上下文的 chunk 语义更完整，向量检索的匹配度更高

#### 🚩 判断标准：

| 回答水平 | 表现 |
|---------|------|
| ✅ 真做过 | 能说出 chunk size 设了多少（通常 512-1024 token），overlap 设了多少（通常 50-200），以及**为什么选这个值**（太大→检索不精准，太小→上下文丢失） |
| ⚠️ 半懂 | 知道概念但说不出具体参数，或者说不清"层级感知"的实现方式 |
| ❌ 背的 | 只会说"比传统的好"，问 chunk size 答不上来 |

#### 🎯 连续追问：

> **"那 40% 这个数字是怎么测出来的？Context Recall 的定义是什么？"**

**答案**：
- **Context Recall**（上下文召回率）= 标准答案中的关键信息，有多少被检索出来的文档覆盖到了
- 计算方式：需要一个标注好的测试集（每个问题对应正确的参考文档），然后看检索出的 Top-K 文档是否包含了这些参考文档
- `Recall = 被覆盖的关键点数 / 标准答案总关键点数`
- 如果他说不清 Context Recall 的定义，那 40% 这个数字大概率是编的

## 🔍 阶段二：检索系统（向量化 + 混合检索 + 重排序）

### Q3：Embedding 选型

> **"你为什么选 Gemini Embedding？和 OpenAI 的 text-embedding-3 或者开源的 BGE 比有什么优势？"**

#### ✅ 好的回答应该包含：

- **Gemini Embedding** 的优势：Google 的模型，中英文都还行，token 上限比较大（最高支持 3072 维度），价格便宜
- **OpenAI text-embedding-3**：效果好但贵，且数据会经过 OpenAI 服务器（内部文档可能有合规问题）
- **BGE（开源）**：可以本地部署不泄露数据，但需要自己维护 GPU 推理服务
- **真正重要的 trade-off**：主要考虑因素是 **成本、数据隐私、效果、维度大小**

#### 🚩 判断标准：

如果他能说出"我们对比了几个模型的检索效果（比如用 MTEB 排行榜或自己的测试集评估）然后选的"，说明是认真选型的。如果只说"领导说用 Gemini 就用了"，那也算诚实。**最怕的是答不出为什么选这个**。

#### 🎯 追问：

> "Embedding 的维度你用的多少？维度越高越好吗？"

**答案**：不是越高越好。维度高 → 存储成本高、检索速度慢；维度低 → 可能丢失语义信息。通常 768 或 1024 是性价比最好的。如果他知道可以通过 **Matryoshka 降维**（俄罗斯套娃降维，新一代 embedding 模型支持的特性）来灵活调整，说明确实研究过。

---

### Q4：混合检索 ⭐（核心考点！答不出基本就是背的）

> **"BM25 + Embedding 混合检索，具体怎么合并两路结果的？权重怎么调？"**

#### ✅ 好的回答应该包含：

**先解释为什么要混合：**
- **BM25**（传统关键词匹配）：擅长精确匹配，比如用户搜"ERROR 404"这种关键词，BM25 很准
- **Embedding**（语义向量检索）：擅长理解语义，比如用户问"页面打不开"，能匹配到"HTTP 404 Not Found"的文档
- 单用任何一个都有盲区，混合起来效果最好

**合并方法（关键知识点）：**
- **方法一：分数归一化后加权求和**（Score Fusion）
  - 把两路的分数分别归一化到 0-1，然后 `final_score = α × BM25_score + (1-α) × embedding_score`
  - α 通常 0.3-0.5，需要调参
- **方法二：Reciprocal Rank Fusion（RRF）**（更常用）
  - 不看分数，只看排名
  - 公式：`RRF_score = Σ 1/(k + rank)`，k 通常取 60
  - 好处：不用归一化，两路分数量纲不同也没关系

#### 🚩 判断标准：

| 回答水平 | 表现 |
|---------|------|
| ✅ 真做过 | 能说出用了 RRF 还是 Score Fusion，知道 α 怎么调的，或者说用了某个框架（如 LangChain 的 EnsembleRetriever） |
| ⚠️ 半懂 | 知道要混合，但说不清合并策略 |
| ❌ 背的 | 只会说"BM25 + Embedding"，问怎么融合就卡住了 |

### Q5：重排序 ⭐（核心考点！）

> **"你提到用 BGE-Reranker 做重排序，能解释一下它和 Embedding 检索（Bi-Encoder）的区别吗？为什么需要多这一步？"**

#### ✅ 好的回答应该包含：

| | Bi-Encoder（Embedding 检索） | Cross-Encoder（重排序） |
|---|---|---|
| **原理** | query 和 document **分别**编码成向量，算余弦相似度 | query 和 document **拼在一起**送进模型，直接输出相关性分数 |
| **速度** | 快 ⚡（可以预计算文档向量，只需算 query 向量 + 向量搜索） | 慢 🐌（每个候选文档都要和 query 一起过一遍模型） |
| **精度** | 一般（因为 query 和 doc 分开编码，交互信息有限） | 高 🎯（两者拼在一起，模型能看到完整的交互关系） |
| **用法** | 第一步：从 10万文档 中快速召回 Top-50 | 第二步：对 Top-50 精排，选出 Top-3 |

**简单类比**：
- Bi-Encoder 就像**粗筛**（海选），速度快但不够精准
- Cross-Encoder 就像**终面评委**，精准但慢，所以只对少量候选人用

#### 🚩 判断标准：

如果他能清楚说出 "Bi-Encoder 编码是独立的，Cross-Encoder 是交叉编码的" 这个核心区别，说明理解了。如果只会说"加了重排序效果更好"但解释不了为什么，那就是背的。

#### 🎯 追问陷阱：

> "BGE-Reranker 的延迟大概多少？对你整体链路的延迟影响有多大？"

**答案**：Cross-Encoder 重排 20-50 条文档，通常需要 100-500ms（取决于模型大小和硬件）。如果他连这个量级概念都没有，说明没有实测过。

## 💬 阶段三：生成与评估（Prompt + Ragas + 幻觉控制）

### Q6：Citation-Aware Prompt

> **"你说实现了 100% 来源溯源，具体 Prompt 怎么设计的？"**

#### ✅ 好的回答应该包含：

- **Citation-Aware Prompt** 的核心思路：在 system prompt 里要求 LLM 每一句回答都标注引用来源
- 典型做法：

```
给你以下参考文档，每段文档有编号 [1] [2] [3]...
请基于这些文档回答用户问题。
规则：
1. 每个观点后必须标注来源编号，如 [1][3]
2. 如果文档中没有相关信息，直接说"我不确定"
3. 不要编造文档中没有的信息
```

- **100% 溯源怎么验证**：让 GPT-4 做 judge，检查回答中的每个 claim 是否都有对应的 citation，且 citation 确实支持该 claim

#### 🎯 追问：

> "如果 LLM 给了错误的引用编号（张冠李戴），你怎么检测？"

**答案**：这正是 Ragas 里 **Faithfulness**（忠实度）指标的作用——它会把回答拆成一个个 claim，然后逐一验证每个 claim 是否能在检索到的上下文中找到依据。如果他能说出这个，说明真正理解了评估体系。

---

### Q7：Ragas 评估体系 ⭐⭐（最核心考点！！）

> **"你简历里提到 Faithfulness 和 Answer Relevance，能具体说说这两个指标怎么算的吗？"**

#### ✅ 好的回答应该包含：

**Faithfulness（忠实度）**：
- 检测 LLM 的回答是否"忠于"检索到的文档（有没有瞎编）
- 计算过程：
  1. 把 LLM 的回答拆成多个独立的陈述（claims）
  2. 对每个 claim，用另一个 LLM 判断：这个 claim 能否从检索到的 context 中推导出来？
  3. `Faithfulness = 有据可查的 claims 数 / 总 claims 数`
- 他说的 95.6% 意思是：100 个陈述里只有约 4 个是文档里找不到依据的

**Answer Relevance（答案相关性）**：
- 检测回答是否真正回答了用户的问题（有没有答非所问）
- 计算过程：
  1. 根据 LLM 的回答，反向生成几个"可能的原始问题"
  2. 把这些生成的问题和真实问题做向量相似度对比
  3. 相似度越高 → 回答越切题

#### 🚩 判断标准：

| 回答水平 | 表现 |
|---------|------|
| ✅ 真做过 | 能准确描述计算过程，知道 Ragas 底层要调用 LLM 来做 judge |
| ⚠️ 半懂 | 只能说"Faithfulness 就是看有没有幻觉"，说不出计算步骤 |
| ❌ 背的 | 把 Faithfulness 和 Accuracy 搞混，或者完全说不出来 |

#### 🎯 致命追问：

> "你说用 GPT-4 做 LLM-as-a-Judge，但 GPT-4 本身也会犯错啊，你怎么保证评估的可靠性？"

**答案**：这是 LLM-as-a-Judge 的经典难题。好的回答应该提到：
- 可以做**人工抽检**（sample 一部分，人工评分和 GPT-4 评分算一致性/Cohen's Kappa）
- 可以用**多个 LLM 交叉评估**（比如 GPT-4 + Claude 双重判断）
- 定期更新 Golden Dataset 覆盖更多 edge case

如果他承认"这确实是个问题，我们做了人工抽检来 calibrate"，加分。如果完全没想过这个问题，扣分。

### Q8：Golden Dataset

> **"你说构建了 200+ 条的 Golden Dataset，这些数据是怎么来的？谁标注的？"**

#### ✅ 好的回答应该包含：

- **来源**：从真实用户的提问日志里筛选的高频问题 + 团队人工补充的边界 case
- **标注内容**：每条包含 `(question, ground_truth_answer, relevant_document_ids)`
- **标注人**：最好是领域专家（比如资深工程师）来标注正确答案和对应的参考文档
- **维护方式**：持续更新，遇到 bad case 就加进去

#### 🚩 判断标准：

如果他说"我自己写的 200 条问答对"——那 ok，说明确实做了。如果说"让 GPT-4 自动生成的"——也行但要追问生成的质量怎么保证。如果含糊其辞——那 200 这个数字可能是编的。

## ⚡ 阶段四：性能与工程化（缓存 + 压测）

### Q9：Semantic Cache

> **"Redis Semantic Cache 是怎么实现的？Cosine Similarity > 0.95 这个阈值怎么定的？"**

#### ✅ 好的回答应该包含：

- **实现原理**：
  1. 用户提问进来，先算 embedding
  2. 拿这个 embedding 去 Redis 里做向量搜索（Redis Stack 支持 VSS）
  3. 如果找到相似度 > 0.95 的历史问题，直接返回缓存的答案，不再走 RAG 链路
  4. 如果没命中，走完 RAG 后把 `(query_embedding, answer)` 存入 Redis

- **0.95 怎么定的**：
  - 太低（如 0.85）→ 语义不同的问题也会命中缓存，返回错误答案
  - 太高（如 0.99）→ 只有几乎一模一样的问题才命中，缓存命中率太低
  - 0.95 是通过在测试集上实验得出的平衡点

#### 🎯 追问陷阱（很重要！）：

> "Semantic Cache 会不会出现语义相近但答案不同的误命中？你怎么处理？能举个例子吗？"
>
> 典型 edge case：`"Redis 的内存淘汰策略是什么"` vs `"Redis 的内存回收机制是什么"`，关键词高度重叠、相似度可能达 ~0.95，但淘汰策略（eviction policy）和回收机制（含过期删除等）其实是不同概念，答案不同。

**答案**：这确实是 Semantic Cache 的一个经典坑。解决方法：
- 加**元数据过滤**（比如结合问题分类/关键词做二次校验）
- 或者在缓存策略里加**时间衰减**（TTL），不长期缓存
- 如果他完全没考虑过这个问题，说明工程思维不够严谨

### Q10：压测数据（灵魂追问）

> **"你说用 Locust 模拟 50 QPS，P95 1.38s。这个 50 QPS 对于内部系统来说是合理的。但我想问：瓶颈在哪里？1.38s 里时间主要花在哪一步了？"**

#### ✅ 好的回答应该包含：

RAG 链路的延迟分解（大致）：

| 步骤 | 典型延迟 |
|------|---------|
| Embedding query | 50-100ms |
| 向量检索 | 10-50ms |
| BM25 检索 | 10-30ms |
| Cross-Encoder 重排 | 100-300ms |
| LLM 生成（最慢！） | 800-2000ms |
| **总计** | **~1-2.5s** |

- **瓶颈明显在 LLM 生成这一步**（占 60-70% 延迟）
- TTFB 0.46s 合理（第一个 token 出来的时间，主要是 LLM 推理的首 token 延迟）
- **链路异步化**主要优化的是：embedding 和 BM25 检索并行执行、SSE streaming 输出

#### 🚩 判断标准：

如果他能清楚说出"瓶颈在 LLM 调用"并且知道每一步大概的延迟量级，说明是亲手压测分析过的。如果只会说"优化后变快了"，那压测数据可能是编的。
